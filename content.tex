\section{Executive summary}

Assignment 3 was based on improving our Catan simulator code by adding three design patterns that would make it more efficient and functional. The first design pattern used in this assignment was the Command Pattern for implementing the undo and redo feature in Catan. With this feature, users can now reverse and redo their moves in the game. We added commands for road, settlement, and city construction, as well as a Command Manager for managing commands. The second design pattern used in this assignment was the Observer Pattern for implementing automatic notification of dice rolling for each player. With this feature, users do not have to manually distribute resources after rolling the dice. The third design pattern used in this assignment was not implemented; it was based on using the Strategy Pattern for separating player decision-making logic from the Player class. With this feature, users can easily switch from human, random AI, and rule-based AI. The machine intelligence for this feature is based on a value system, where actions that give victory points are given the highest priority, followed by non-victory point buildings, and finally constraints such as spending cards if a player has more than seven cards. Overall, the assignment demonstrated how design patterns can significantly improve software architecture by promoting loose coupling, single responsibility, and open/closed principles.

\section{Design pattern 1}

\textbf{Which design pattern did you choose and why?}

We chose the Command Pattern because it lets us create each player action like building a road, city, or settlement as an object. This made it easy to support the undo and redo functionality, as each command can store the information needed to redo their actions. 

Instead of calling methods like buildSettlement() in TurnManager, we made these actions into Command classes. It made the code simpler as they can be executed, undone, and redone. Furthermore, it improves scalability. If new actions need to be added in the future, they can be implemented as new command classes without changing existing code.
\\

\textbf{Explain the design pattern and elaborate on how it fosters cleaner design in this
particular software.}

The Command Pattern is a design pattern where each action in the program is represented as an object.

\begin{itemize}
\item \textbf{Command:} Command interface with execute(), undo(), and redo() functions
\item \textbf{Command Classes:} BuildRoadCommand, BuildSettlementCommand, and BuildCityCommand
\item \textbf{Invoker:} CommandManager - stores and manages the command history
\item \textbf{Receiver:} TurnManager - calls the undo/redo action on the respective command
\end{itemize}

This helped make the design much cleaner because it separates what action is being done from how the system manages those actions over time. For example, building a road or upgrading a city is handled inside its own class, and the TurnManager class which uses the undo/redo system doesn’t need to know how those actions work internally, but rather it just calls undo() or redo() on the command. This avoids having large, messy logic inside those classes, which also makes the code more organized and easier to extend. If we want to add a new type of action in the future, we can just create a new command class instead of modifying a lot of the existing code. This follows good design principles like keeping classes focused on one responsibility and making the system easier to maintain.\\\\
Overall, the Command Pattern works really well for undo/redo because it naturally allows us to store actions and reverse them, and makes the implementation more aligned with the behavior we wanted from the system.


\textbf{How much did you need to think about the structure of the evolved program once you have figured out which design pattern to use?}

Once we figured out that the Command pattern was the right fit, we did not need to completely rethink the whole structure of the evolved program, but we still had to think carefully about where each responsibility should go. We knew we needed a command interface, command classes for each action, and the turn manager to call all the undo/redo operations. So in that sense, the design pattern made the structure easier to organize because it gave us a clear template to follow. Instead of putting all the logic inside one large controller or game class, we could break the system into smaller pieces with specific roles.\\\\
However, we still had to think about some details regarding the structure of the program. We still had to think carefully about which actions should be turned into commands, what data each command needed to store in order to undo correctly, and how the manager should track the current command position. This was due to the fact that some commands require remembering previous values or board states so that undo can accurately restore the system which required careful planning. For example, in BuildRoadCommand, we had to decide that the command should store the Board, Player, and the two node IDs. Without that information, the command would not be able to properly execute, undo, or redo the road-building action.\\\\
Overall, once we identified the pattern, the structure of the program became much clearer and allowed us not having to figure everything out from scratch, but rather focus on smaller details, such as what each command should store for their undo/redo functions. So the pattern made things simpler by removing a lot of complexity, but we still needed to plan the design carefully to properly execute the pattern.\\

\textbf{How much change was needed in the design and in the implementation?}

We made changes that were required in the design and implementation of the undo/redo functionality. Initially, player actions were handled in TurnManager. To apply Command Pattern, we had to refactor these action methods into separate command classes. 

We also added a CommandManager class that would keep track of all executed commands and manage the undo/redo operations. This needed us to think about how the actions would be called during a player’s turn.

In addition, we had to modify the Board and Player classes to help with undoing actions. We had to implement methods to remove buildings and roads and make sure that the player got those resources back.

Although these changes required restructuring the code, the core logic of the game was the same. The changes improved the organisation of the code and helped make it easy to maintain.

\section{Design Pattern 2}

\textbf{Which design pattern did you choose and why?}

We chose the Observer Pattern for handling dice roll notifications in the Catan game. 
In Catan, when dice are rolled, multiple tiles need to react by producing resources.
This creates a one to many dependency: one MultiDice object and many Tile objects.
The Observer pattern perfectly captures this relationship where when one object changes state, all its dependents are automatically notified and updated.
It decouples the dice from the tiles, the dice doesn't need to know which tiles exist or how they produce resources. 

\textbf{Explain the design pattern and elaborate on how it fosters cleaner design in this
particular software.}

The Observer Pattern defines a one to many dependency between objects so that when one object changes state, all its dependents are notified and updated.

\begin{itemize}
    \item Subject: DiceRollSubject abstract class extends MultiDice
    \item Observer: DiceRollObserver interface implemented by Tile
    \item Notification: When MultiDice.roll() is called, it calls notifyObservers(diceValue)
\end{itemize}

\textbf{Cleaner design:}
\begin{itemize}
    \item Loose Coupling: MultiDice doesn’t need to know anything about Tile, it just knows it has observers. Tiles can be added or removed without modifying the dice class (but in this game we don't need to remove the tile). New observer types could be added later that require dice roll. 
    \item Open/Closed Principle: We can add new observers without modifying the subject. The system is open for extension but closed for modification. 
    \item Single Responsibility: MultiDice only handles rolling dice and notification. Tile only handles resource production logic. Board class only handles building logics. Each class had one reason to change. 
    \item Simplified Communication: Without Observer, we would need to manually call each tile’s method after every dice roll. The pattern automates this communication. 
    \item Tiles register themselves with dice at runtime in Board.registerTileWithDice(). We could add or remove tiles if needed. 
\end{itemize}



\textbf{How much did you need to think about the structure of the evolved program once
you have figured out which design pattern to use?}

Moderate thought was needed: 

\begin{itemize}
    \item Initial Analysis: 
    \begin{itemize}
        \item We identified that dice rolls affect multiple tiles.
        \item We realized that is a classical observer scenario.
        \item We decided on the observer pattern after some discussion.
    \end{itemize}

    \item Planning: 
    \begin{itemize}
        \item We determined what should be the subject (MultiDice), observer (Tile) and notification method (onDiceRolled(int diceValue). 
        \item In this case Board class was the best choice for registration as it has references to Tile, robber and other classes needed. 
    \end{itemize}

    \item Other: 
    \begin{itemize}
        \item We had to consider how the robber blocks production (Dice == 7). 
        \item Needed to pass dependencies to tiles ( like robber, logger)  and ensured tiles only produce when their token matches the dice value.
    \end{itemize}
\end{itemize}

\textbf{How much change was needed in the design and in the implementation?}

Moderate changes were required: 

\begin{itemize}
    \item Added new classes: DiceRollSubject (abstract class), DiceRollObserver (interface) 
    
    \item Modified existing classes: 
    \begin{itemize}
        \item MultiDice: added extending DiceRollSubject, and in roll method notifyObserver(sum) is called. 
        \item Tile: Added implements  DiceRollObserver and implements onDiceRolled() , distributedResources(), distributeResourceToPlayer(), and shouldSkipTile().
        \item Board: Added registerTileWithDice() method and removed all resource production methods that previously existed (Relocating these methods to Tile class).
    \end{itemize}
\end{itemize}

\section{Design pattern 3}

\textbf{Which design pattern did you choose and why?}

We chose the Strategy Pattern to handle how different players decide their actions during a turn. In our original design, we had separate classes like HumanPlayer and ComputerPlayer, which meant the behavior for choosing actions was tied directly to the player type, which made the design less flexible and harder to extend.

By using the Strategy Pattern, we moved the decision-making logic into separate strategy classes, such as HumanTurnStrategy, RandomTurnStrategy, and RuleBasedStrategy. By using this design, each player now can use a strategy object that determines how they choose their next command. This allows us to easily switch or add new behaviors without changing the Player class.

We chose this pattern because player behavior is something that can vary a lot (human input, random input, rule-based AI), and the Strategy design pattern is designed exactly for that situation. It also improves the design by following principles like Single Responsibility (player stores data, strategy handles decisions) and Open/Closed (we can add new strategies without modifying existing code) principles.
\\\\\\

\textbf{Explain the design pattern and elaborate on how it fosters cleaner design in this
particular software.}

The strategy pattern lets us design the different ways players can take a turn, and lets us switch easily between them. We created a TurnStrategy interface, and different classes such as HumanTurnStrategy, RandomTurnStrategy, and RuleBasedStrategy to implement it.

This makes the design better because instead of putting all the logic in Player, we move it to different Strategy classes. It makes the code clean and easier to understand, since every class has one responsibility. It also makes the code scalable, as we can easily add a new strategy without changing existing code. We can just add a new class that implements TurnStrategy.
\\

\textbf{How much did you need to think about the structure of the evolved program once you have figured out which design pattern to use?}

After we decided to use the Strategy design pattern, we knew that all-turn related logic would be moved to separate classes, and the Player class would use them instead of having the logic itself. After that, we had to organize the classes correctly. We came up with an interface TurnStrategy, and made sure that all the strategy classes had the chooseCommand method. We also had to update the Player class to have a reference to a strategy and call it in a turn. 

We had to do some planning to decide what logic would be in each strategy, but the pattern helped us to make that decision. This made it easy to implement new features, like when implementing the machine AI requirement.
\\

\textbf{How much change was needed in the design and in the implementation?}

The change required in the design was moderate, as we had to remove the inheritance-based structure (HumanPlayer and ComputerPlayer extends Player) and replace it where Player contains a TurnStrategy object instead. This required updating the UML diagram to introduce the strategy interface and its implementations, and modifying relationships between classes.

Inside of the Player class, we had to refactor how turns are handled. Instead of calling something like takeTurn() directly on different player subclasses, we introduced a method like chooseCommand() in the Player class that delegates choosing the command to the strategy object instead of being directly in the Player. This meant moving input handling and AI logic out of the player classes into their respective strategy classes.

Although we didn't implement this pattern in the code but rather only in the design, the changes would be fairly localized to player behavior and turn handling. Once the strategy structure is set up, the rest of the system (like TurnManager and Game) becomes simpler as it does not need to care if a player is human or a computer anymore to get the current action in their turn.
